# 死亡背包與 `/back`

玩家死亡時，模組會在原版將物品生成為世界掉落物之前，直接從玩家的 Server 權威物品欄、裝備與玩家持有的暫存介面資料建立死亡背包。一般物品會被封裝進死亡背包；DeadRecall 的一般背包與死亡背包本身維持獨立，生成為世界掉落物，避免背包巢狀。

## 行為

- 擷取玩家主要物品欄、快捷列、盔甲、副手、目前介面的游標物品，以及玩家固定 2×2 合成格輸入。
- 箱子等外部持久容器的實際儲存槽不屬於玩家死亡背包，不會被讀取或修改。
- ItemStack 直接複製到 Container Data Component，保留數量、耐久、自訂名稱及其他相容 Components。
- 一般物品不會先生成 ItemEntity，因此不會與地面物品合併、被其他玩家撿走或先被環境破壞。
- 容量依物品數動態計算，最多 54 格。
- 拾取延遲為 40 ticks（2 秒）。
- 不會自然消失，並免疫一般破壞。
- 低於虛空傷害線時會取消移除、獲得向上動量並限制下墜。
- 地面上的死亡背包會顯示紅色定位光柱。
- 每個死亡背包包含唯一資料，避免物品實體互相合併。

若玩家沒有可封裝的一般物品，不會建立新的死亡背包；排除的 DeadRecall 背包仍會獨立掉落。游標或 2×2 合成格中的 DeadRecall 背包也不會被放入另一個背包。

## 死亡 Space Unit

死亡背包成功建立後，Totem Nexus 會在死亡位置建立一個 `DEATH` 類型 Space Unit，並自動標記為該玩家已探索。

目前這是死亡節點基礎整合：節點可以進入 Space Unit 資料與地圖過濾流程，但死亡節點回收、隱藏、完整傳送成本與生命週期管理仍未完成。詳見 [Totem Nexus 實作狀態](../nexus/status.md)。

## `/back`

模組暫存玩家最近一次死亡的維度與座標。執行 `/back` 可跨維度返回；成功使用後會清除記錄，避免重複傳送。

`/back` 與 Totem Nexus 死亡節點目前是兩套並存機制：`/back` 使用暫存死亡位置；Nexus 使用世界 SavedData 的 Space Unit 記錄。

## 安全條件

- 只有 Server 在原版 `Inventory.dropAll()` 前決定擷取內容。
- 原版 `keepInventory=true` 時不會執行死亡背包擷取。
- Inventory／Equipment 的消失詛咒由原版先處理；游標與 2×2 合成格中的消失詛咒物品由同一個 Server transaction 補上相同行為。
- DeadRecall 背包不會放進另一個死亡背包。
- 建立過程發生例外時，模組會丟棄未完成的死亡背包、停用未完成的死亡節點，並把游標／合成格物品安全回填到 Inventory，讓原版掉落流程接手。
- 舊的死亡點半徑掃描器已完整刪除；無論成功或失敗，都不會掃描或回收附近 ItemEntity。
- Discord 或玩家通知失敗不會撤銷已經完成的死亡背包交易。
- crafting table、anvil、smithing table 等外部工作站自己的暫存輸入槽，以及第三方飾品槽仍待後續相容性確認。
- 背包介面固定追蹤開啟時的 ItemStack，避免資料回注到後來換手的其他物品。
