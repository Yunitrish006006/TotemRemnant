# Totem Nexus：自適應多方塊傳送陣設計備忘

> 狀態：Future / Deferred
>
> 此文件只記錄未來把 Totem Nexus 從 DeadRecall 拆成獨立模組後的設計方向，不納入目前 DeadRecall 的實作與發行範圍。正式開發前必須另建 OpenSpec change，重新確認數值、相容性、遷移與測試需求。

## 1. 設計目標

傳送陣不應只是單一功能方塊或固定工作台配方，而應是一座會隨目的地數量、距離、維度與負載逐步擴建的世界內多方塊結構。

主要目標：

- 優先使用 Minecraft 原版方塊，不新增大量專屬材料。
- 鼓勵玩家使用傳送陣所在地附近可取得的材料。
- 鼓勵材料多樣性，避免所有傳送陣使用同一組最高效率方塊。
- 每條目的地連線都形成可辨識的結構負載，而不是只在 GUI 增加一列。
- 傳送陣規模應隨支援度增加而擴大，讓大型樞紐自然成為大型建築。
- 結構判定必須 Server authoritative、可持久化、可診斷，且不可依賴每 tick 全量掃描。

## 2. 傳送陣身分與歸屬

每座傳送陣由一個唯一核心定義。核心保存永久 `networkId`，外部功能部件需綁定同一 `networkId`，並通過位置及結構驗證後才提供能力。

判定分成三層：

1. **身分判定**：部件與核心的 `networkId` 相同。
2. **歸屬判定**：部件已登記到該核心，且核心仍存在。
3. **有效性判定**：部件位置、環帶、連通性、對稱性與距離符合規則。

因此部件可以「屬於某座傳送陣，但目前無效」，GUI 必須顯示具體失效原因。

建議資料：

```java
record NexusRecord(
    UUID networkId,
    UUID ownerUuid,
    ResourceKey<Level> dimension,
    BlockPos corePos,
    int structureTier,
    Set<PartReference> parts,
    Set<ConnectionReference> connections
) {}
```

核心、部件與全域 registry 必須能處理區塊未載入、伺服器重啟、核心拆除及舊資料遷移。

## 3. 四項結構能力

每座傳送陣由原版方塊與其結構位置提供四項能力。

### 3.1 連結力（Link Capacity）

決定可維持的目的地連線數、複雜連線及同時啟用的路線數。

適合來源：

- 紅石線、紅石火把、紅石塊。
- 中繼器、比較器。
- 銅方塊、切製銅、避雷針。
- 終界燭等具有空間連結語意的原版方塊。

### 3.2 擴張力（Expansion Capacity）

決定核心可控制的半徑、環帶數、結構方塊上限及可配置的共鳴位置數。

適合來源：

- 石磚、深板岩磚、黑石磚。
- 鐵塊、銅塊、石英磚。
- 黑曜石、哭泣的黑曜石。

擴張力必須主要來自合法骨架、環帶與支柱，不得單純計算附近堆積方塊。

### 3.3 準確力（Precision）

決定目的地鎖定、落點偏差、方向保持、安全落點搜尋及跨維度校準能力。

適合來源：

- 紫水晶塊、紫水晶芽、晶簇。
- 比較器、目標方塊。
- 幽匿感測器、校頻幽匿感測器。
- 石英、磁石、終界箱。

準確力不足時優先增加準備時間、限制啟動或在安全區內偏移；不得把玩家隨機丟入岩漿、虛空或碰撞方塊。

### 3.4 傳送力（Transfer Power）

決定最大距離、跨維度能力、單次實體負載、坐騎／物品支援、冷卻時間及能量需求。

適合來源：

- 紅石塊、岩漿塊、螢石、海燈籠。
- 黑曜石、哭泣的黑曜石。
- 重生錨、烽火台。

高價值或唯一方塊不得成為普遍必要條件。龍蛋、終界水晶等只能作為可選特性或特殊伺服器規則，不得成為標準 progression 的硬需求。

## 4. 方塊能力資料驅動

方塊能力不得硬編碼在大量條件分支中。應使用資料驅動 registry 或 datapack 可覆寫設定：

```json
{
  "block": "minecraft:copper_block",
  "attributes": {
    "link": 3,
    "expansion": 2,
    "precision": 0,
    "transfer": 1
  },
  "allowed_zones": ["conduit", "frame"],
  "local_resonance_tags": ["copper"],
  "diminishing_group": "copper_blocks"
}
```

同一方塊可提供多項能力，但不同結構位置應套用不同倍率。

建議結構區域：

- `core`：校準與定位，偏向準確力。
- `conduit`：由核心向外的導線，偏向連結力。
- `frame`：環帶、骨架與支柱，偏向擴張力。
- `power`：外圈動力柱或能源節點，偏向傳送力。
- `destination_anchor`：每個目的地專屬的共鳴柱或錨點。

隨意埋在附近、未綁定或不在合法位置的方塊不得計分。

## 5. 目的地負載

每條連線不是固定消耗一個插槽，而是產生四項負載：

```java
record ConnectionLoad(
    double linkCost,
    double expansionCost,
    double precisionCost,
    double transferCost
) {}
```

負載至少受以下因素影響：

- 已啟用目的地總數。
- 同維度實際距離。
- 跨維度類型。
- 目標節點類型與穩定度。
- 單次允許的玩家／實體負載。
- 是否允許坐騎、物品或容器傳送。
- 來源與目標傳送陣的結構能力。

所有啟用連線的負載會累加。當能力不足時，系統應停用超出負載的連線或拒絕新增連線，不得靜默降級既有安全條件。

## 6. 逐步擴建與目的地結構

每個目的地應對應一個世界內可見的 `destination_anchor`，例如共鳴柱、外圈節點或放射狀分支。

建議規則：

- 第一層環帶支援少量本地連線。
- 第二層環帶增加目的地位置與中距離能力。
- 第三層以上支援大型樞紐、長距離及跨維度連線。
- 新增目的地時，玩家必須新增或啟用一個對應錨點。
- 遠距離與跨維度目的地可要求更外層、更高或更完整的結構。
- 來源與目標兩端都必須達到路線最低能力，避免單端大型樞紐完全替代目標端建設。

結構規模不應只由抽象 Tier 決定；Tier 應是掃描結果的摘要，而不是唯一判定來源。

## 7. 對稱性與結構效率

對稱性應作為效率與穩定倍率，而不是所有造型的硬性開關。

可考慮：

- 四向對稱：最高效率。
- 二向鏡像：中度加成。
- 接近對稱：無加成也無懲罰。
- 嚴重偏置：降低準確力或傳送力。

不同能力可採不同結構指標：

- 連結力：放射分支分布與連通性。
- 擴張力：環帶完整度、骨架連續性與支柱分布。
- 準確力：核心區鏡像與目的地錨點成對校準。
- 傳送力：動力柱高度、分散度與負載平衡。

所有判定以核心為原點，使用相對座標與旋轉／鏡像規則，不依賴絕對世界座標。

## 8. 當地材料共鳴

傳送陣第一次建立或重新校準時，可根據所在地的 Dimension、Biome、高度與有限範圍採樣，建立當地材料 profile。

原則：

- 當地材料提供加成，不作為絕對必要條件。
- 採樣必須有方塊 Tag 白名單與最大樣本數。
- 禁止遍歷整個區塊群或每 tick 掃描。
- 不應選取基岩、容器、生成器、指令方塊、流體或其他模組機器作為要求。
- 建議採弱來源限制：材料在當地自然存在即可，不追蹤每個 ItemStack 的採集來源。

可依當地 profile 對原版方塊套用倍率，例如當地石材增加擴張力、當地銅增加連結力、當地紫水晶或幽匿系列增加準確力。

## 9. 材料多樣性與遞減收益

為避免單一最佳材料壟斷，必須同時採用：

1. **同類材料遞減收益**：超過門檻後逐段降低計分。
2. **材料多樣性倍率**：使用多個有效材料群組時提高整體效率。
3. **區域／位置限制**：高分方塊必須位於合法結構位置。
4. **稀有材料上限**：高階方塊提供強力加成，但不可無限堆疊。

示例遞減：

- 前 8 個：100%。
- 第 9–16 個：50%。
- 第 17–32 個：25%。
- 超出上限：不計分。

實際門檻應由 datapack 設定，並以材料群組而非單一 block ID 計算，避免不同氧化銅或裝飾變體繞過限制。

## 10. 掃描與效能

結構掃描只能在以下事件觸發：

- 首次註冊。
- 綁定或解除部件。
- 已登記範圍內的方塊變更。
- 手動校準。
- 傳送前驗證。
- 世界載入後首次使用。
- 結構受損或核心狀態改變。

要求：

- 禁止每 tick 全量掃描。
- 掃描必須限制半徑、方塊數、深度與 chunk 數。
- 大型結構應使用 dirty region、快取與增量更新。
- 未載入 chunk 不得由 Client 宣稱有效；Server 可使用已保存快照並在真正傳送前要求必要區塊驗證。
- 兩座核心控制區域重疊時必須有確定性歸屬與衝突診斷。

## 11. GUI 與診斷

管理介面至少顯示：

```text
連結力：目前值 / 已用負載
擴張力：目前值 / 已用負載
準確力：目前值 / 路線最低需求
傳送力：目前值 / 已用負載
```

並列出：

- 各目的地的四項負載。
- 哪些結構部件已綁定但無效。
- 對稱性、環帶完整度與材料多樣性倍率。
- 同類材料遞減造成的損失。
- 當地材料共鳴加成。
- 建議補強的原版方塊種類與合法位置。

Client 只能顯示 Server 計算結果，不得提交最終能力、負載或結構有效性。

## 12. 與目前 DeadRecall 的邊界

在 Totem Nexus 正式拆分前：

- 現有磁石 Space Unit、傳送成本、安全落點及穩定度規格維持有效。
- 本文件不要求目前版本改造為完整自由多方塊網路。
- 不新增大量專屬建材或材料物品。
- 不改變既有世界資料 identifier 與已註冊節點語意。
- 可先保留結構快照、Tag 化材料與 Server authoritative 掃描，作為未來遷移基礎。

正式開發時應另行決定：

- 核心是否仍使用磁石，或改為獨立核心方塊。
- 舊磁石節點如何升級或轉換。
- 自由造型與固定環帶模板的平衡。
- 方塊能力初始數值與 datapack schema。
- 目的地錨點的建造、拆除與雙端握手流程。
- 管理員旁路、伺服器設定及破壞保護整合。
- GameTest、效能測試、世界遷移與故障恢復矩陣。
